# パッシブ致死 構想書

> **ステータス**: 実装済み
> **着想元**: `doc/終了済み/ReEncountステータス変化構想.md` §2.3 から切り出し

## 1. 概要

歩行消化（`ReEncountCallback` 内の `AllPassiveWalkEffect` ループ）によって敵のHPが0以下になった場合に、
エンカウントせずにログ表示する仕組み。

戦闘で直接倒すのではなく、**かつて付与したパッシブが時間差で敵を殺す** という状況への対応。

## 2. パッシブ致死ログ

### 内容
- 歩行消化の結果、敵のHPが0以下になった場合にログ表示
- 「○○のパッシブにより△△を倒した」旨を表示
- 複数パッシブで倒された場合はランダム順で判定し、HPを0以下にしたパッシブの付与者の手柄

### 表示先

エンカウント時の状況に応じて使い分ける。

| ケース | 状況 | 戦闘 | 表示先 |
|--------|------|------|--------|
| ソロ敵が致死 | 1体のみ、パッシブで死亡 | 入らない | MessageDropper（歩行ログ） |
| グループの一部が致死 | 生存者がいるため戦闘開始 | 入る | SchizoLog（戦闘開始時） |
| グループ全員が致死 | 全員死亡 | 入らない | MessageDropper（歩行ログ） |
| 友情コンビの片方が致死 | 相方は生存、戦闘開始 | 入る | SchizoLog（戦闘開始時） |
| 友情コンビの両方が致死 | 全滅 | 入らない | MessageDropper（歩行ログ） |

- **戦闘に入らない場合**: MessageDropperで歩行中に淡々と通知。歩いていたら敵が勝手に死んでいた演出
- **戦闘に入る場合**: SchizoLogで戦闘開始時に「○○は○○により倒れていた」旨を表示

### 現状との差分
現在の `ReEncountCallback` では、歩行消化後の死亡判定で復活準備（`PrepareReborn`）に入るのみ。
「どのパッシブの付与者が殺したか」の追跡やログ出力は行っていない。

### 実装上の前提：Callback後の再フィルタが必要

現在の `EncounterEnemySelector.Select()` では、`FilterEligibleEnemies` が最初に1回だけ実行され、
その後の `ApplyReencountCallbacks` でパッシブ歩行効果によりHP≤0になった敵は **validEnemiesリストから除去されない**。

```
現在のフロー（問題あり）:
  FilterEligibleEnemies → validEnemies（生存者のみ）
  ApplyReencountCallbacks → ここでパッシブにより死亡する敵が出うる
  ApplyComboPassiveAccumulation
  if (!validEnemies.Any()) return null  ← 死体入りリストはすり抜ける
  SelectLeader → 死んだ敵がリーダーに選ばれうる
```

パッシブ致死を実装する際は、`ApplyReencountCallbacks` の直後に死亡敵を再フィルタし、
その際に致死ログ対象を収集する処理を挿入する必要がある。

```
必要なフロー:
  FilterEligibleEnemies → validEnemies
  ApplyReencountCallbacks → パッシブ歩行効果でHP≤0になりうる
  ★ 死亡敵の再フィルタ + 致死ログ対象の収集
  ApplyComboPassiveAccumulation
  if (!validEnemies.Any()) → 全滅ならMessageDropperでログ表示、return null
  SelectLeader → 生存者のみからリーダー選出
  戦闘開始 → 致死ログ対象があればSchizoLogで表示
```

## 3. リエンカウント周りの潜在的問題

パッシブ致死の実装に際して発見された、既存リエンカウントシステムの潜在的問題。

### 3.1 パッシブ蓄積シミュレーション中の死亡ガードなし【高】

`AccumulateComboPassives` のループ内でパッシブの `WalkEffect()` により敵が死亡しても、
`Death()` チェックがなくループが最後まで続く。死んだコンボ相方が `validEnemies` に残る。

- **箇所**: `NormalEnemy.cs` AccumulateComboPassivesのシミュレーションループ
- **§2の再フィルタ問題と連鎖する**: Callback後の再フィルタで同時に解決可能

### 3.2 巨大distanceTraveledのパフォーマンス【中】

`ReEncountCallback` 内の歩行消化ループは `distanceTraveled` 回実行される。
1000歩なら全パッシブ×1000回のループ。終盤のパッシブ数増加で遅延の可能性がある。

- **箇所**: `NormalEnemy.cs` ReEncountCallback内の歩行消化forループ
- **対策案**: ループ中に `Death()` で早期脱出 / distanceTraveledの上限設定 / バッチ処理化

### 3.3 brokenフラグの永続化漏れ【中】

`broken`（オーバーキルによる完全破壊）は `[NonSerialized]` であり、非コンボ敵では永続化されていない。
セーブ/ロード問題に帰結するため、詳細は → `doc/セーブロード関連問題.md` §1

### 3.4 解決の見通し

| 問題 | 対応 |
|------|------|
| §2の再フィルタ漏れ | `ApplyReencountCallbacks` 後に `validEnemies.RemoveAll(e => e.Death())` |
| 3.1 蓄積中の死亡 | 上記の再フィルタで同時に解決。ループ内ガードは任意 |
| 3.2 パフォーマンス | ループ内 `if(Death()) break;` で早期脱出。根本対策は別途検討 |
| 3.3 broken | 調査の上で方針決定が必要 |

## 4. 個別撃破報酬

### 結論：物質的報酬なし（ログのみ）

検討の結果、個別撃破に対する物質的報酬は**設けない**。

パッシブ致死ログの目的は **パッシブの残存性をプレイヤーに伝えること** であり、
エコシステムが勝手に回っている事実が淡々と通知されること自体が体験の核。

### 棄却した報酬候補

| 候補 | 棄却理由 |
|------|---------|
| HP回復・MaxHP成長 | HP固定が設計原則（`doc/HP仕様書.md` §1）。HP報酬は思想に反する |
| 十日能力の成長ボーナス | スキル使用時成長＋戦闘勝利ブーストで既に十分。屋上屋を架す |
| スキル解放進行 | スキル入手は歩行→場所→状態→イベントの連鎖で得る設計。ポイント交換型の解放はこの思想を壊す |

### Reborn敵の扱い
通常死亡と同じ。ログを出す。その後は既存の復活フロー（`PrepareReborn`）に乗る。
