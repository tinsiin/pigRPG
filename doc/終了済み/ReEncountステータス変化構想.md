> **ステータス**: ✅ **全項目完了** — 実装済みの仕様は `doc/ReEncount仕様書.md` / `doc/エンカウント仕様書.md` を参照。§2.3 パッシブ致死ログは `doc/パッシブ致死・撃破報酬構想.md` に切り出し済み

## 1. 概要

再エンカウント時に、**AIの意志とは無関係に**、前回の遭遇からの経過歩数に応じて敵のステータスが変化する仕組み。

戦闘後AIが「敵が自分の意志で行動した結果」であるのに対し、
本システムは **「時間経過による自然な変化」** という別レイヤーを担う。

### 既存システムとの関係

| システム | 性質 | タイミング |
|----------|------|-----------|
| 戦闘後AI（PostBattleActRun） | 敵の意志による行動（回復・蘇生等） | 戦闘終了直後 |
| 戦闘後相性値変動（PostBattleBondCalculator） | 戦闘内行動からの相性値計算 | 戦闘終了直後 |
| 既存パッシブ歩行消化（ReEncountCallback） | 既に持っているパッシブのWalkEffect発火＋残存カウンタ減少 | 再エンカウント時 |
| **本システム（ReEncountステータス変化）** | **AI無関係の機械的な自然変化** | **再エンカウント時** |

## 2. 構成要素

### 2.1 HP自然回復 ✅ 実装済み（友情コンビボーナス含む）

→ 仕様は `doc/ReEncount仕様書.md` §2 を参照

- 通常: `MaxHP × Random(0, recoveryRatio)`
- 友情コンビ: `MaxHP × Random(lowerRatio × recoveryRatio, recoveryRatio)`
  - `lowerRatio = min(ReEncountCount / BondMatureCount, 1.0)` — §2.4の共通計算式を参照
  - 付き合いが長いほど回復の下限が上がり、確実に回復するようになる

### 2.2 再遭遇時パッシブ生存判定 — ✅ 既存実装で解決済み

> **出典**: メモ「パッシブ.md」§敵の歩行ターンによる生存判定
> **結論**: 既存の歩行消化システム（`AllPassiveWalkEffect` + `UpdateWalkAllPassiveSurvival` ループ）が、このメモの構想をより正確な形で実現している。

メモの元々の構想は「残存ターン÷2、効果×1/3で一括適用して消す」という **簡易的なパッシブ消化処理** だった。
これはメモ執筆時点でまだ歩行消化の仕組みが存在しなかったため生まれた発想であり、
現在の `ReEncountCallback` 内の歩数分ループ（1歩ずつ `WalkEffect` 発火 + `DurationWalkCounter` 減少）が
同じ目的をより精密に果たしている。

したがって、この構想は **既存実装によって解決済み** とする。

### 2.4 パッシブ蓄積（友情コンビ専用） ✅ 実装済み

→ 仕様は `doc/ReEncount仕様書.md` §3 を参照

友情コンビのメンバーが、エンカウント間の時間に **お互いにパッシブをかけ合っていた前提** でステータスに反映する仕組み。

#### 基本ルール
- 友情コンビが「ポイント等のコスト計算なしに」有効なスキルでパッシブを付与し合っていた、という前提
- 効果量は **実際に歩いた歩数でキャップ** される
  - 10歩しか経っていないなら10歩分の効果しか乗らない（過剰バフ防止）
- **ReEncountを繰り返すたびに下限が上がる**
  - ReEncountCountが多い＝付き合いが長い＝より確実にパッシブが残っている

#### スキル選定ロジック（既存プロパティで判別可能）
相方の持つスキルから「無条件で味方にかけて弊害がないもの」を抽出:
1. `SkillType.HasFlag(addPassive)` — パッシブ付与スキルである
2. `SkillZoneTrait.HasFlag(SelectOnlyAlly)` or `SelfSkill` — 味方向けである
3. 付与されるパッシブの `IsBad == false` — デバフではなくバフである
4. `CanApplyPassive()` — 既存の適合条件（OkType/OkImpression）で自動フィルタ

#### 処理フロー

```
EncounterEnemySelector.Select():
  ├── ReEncountCallback（各敵ごと）
  │     ├── 既存: 歩行消化ループ（distanceTraveled回）  ← 既存パッシブの消化
  │     ├── 既存: ApplyGrowth
  │     └── 既存: NaturalHPRecovery
  │
  └── ApplyComboPassiveAccumulation（ReEncountCallback の後）
        │   ※ ReEncountCallback内ではなく外。相方インスタンスへのアクセスが必要なため
        │
        各コンボの生存メンバー間で双方向に:
        ├── 1. 相方のスキルから対象パッシブ付与スキルを全て列挙
        ├── 2. 全て付与（ポイント無視）
        ├── 3. simulatedSteps = Random(lowerRatio × distanceTraveled, distanceTraveled)
        │       「相方がsimulatedSteps前にパッシブをかけた」前提
        ├── 4. 新規付与パッシブのみを対象に、simulatedSteps分の歩行消化を自前実行
        │       （WalkEffect + UpdateWalkSurvival を simulatedSteps回）
        └── 5. 残存したパッシブだけが戦闘に持ち込まれる
```

#### 既存の歩行消化との関係

既存の歩行消化ループ（distanceTraveled回）は **既に持っているパッシブ** を消化する。
パッシブ蓄積は **その後** に実行し、新規付与パッシブのみを自前ループで消化する。

これにより:
- 既存パッシブ: 歩行消化で自然に減る（減らす側）
- 新規パッシブ: 蓄積で新しく乗る（増やす側）
- → **パッシブの新陳代謝** が実現する

自前ループの対象は **新規付与パッシブのみ** に限定。
既存パッシブは既に消化済みなので、巻き込むと二重消化になる。

#### ReEncountCount連動の共通計算式

自然回復の友情ボーナスとパッシブ蓄積の両方で使う共通の下限計算:

```
lowerRatio = min(ReEncountCount / BondMatureCount, 1.0)
```

| 定数 | 値 | 意味 |
|------|----|------|
| `BondMatureCount` | **11** | この回数再会したら付き合いが成熟し、下限が上限に追いつく |

| ReEncountCount | lowerRatio | 効果 |
|---------------|------------|------|
| 1回目 | 0.09 | ほぼランダム（恩恵わずか） |
| 3回目 | 0.27 | そこそこ安定 |
| 6回目 | 0.55 | 半分以上は確実 |
| 11回目以降 | 1.00 | 完全成熟（上限と同じ＝確実に恩恵を受ける） |

この式は以下の2箇所に適用される:
- **HP自然回復**: `MaxHP × Random(lowerRatio × recoveryRatio, recoveryRatio)`
- **パッシブ蓄積**: `simulatedSteps = Random(lowerRatio × distanceTraveled, distanceTraveled)`

### 2.5 エンカウント頻度の偏り防止 ✅ 実装済み

→ 仕様は `doc/エンカウント仕様書.md` §1 を参照

生存敵が少ないときにエンカウント頻度を自然に下げる。

- `aliveRatio = 生存敵数 / 全敵数`（線形）をエンカウント判定の乗数に適用
- 既存の pity システムにより、生存敵がいる限り「永遠に会えない」状況は防がれる（全滅時は停止）
- 実装: `GameContext.GetAliveRatio()` + `EncounterResolver.Resolve` の combinedMultiplier に乗算

### 2.6 友情コンビの死亡メンバー支援 ✅ 実装済み

→ 仕様は `doc/ReEncount仕様書.md` §4 を参照

友情コンビのメンバーが死亡している場合、相方の存在によって復活が有利になる仕組み。
戦闘後AIの「意志による蘇生」とは別の、**友情コンビという関係性そのものから来る汎用的な恩恵**。

#### a. 相方のDeathHealによる即時蘇生

友情コンビの相方が `DeathHeal` スキルを持っている場合、
再エンカウント時に **Rebornを待たずに即座に蘇生** する。DeathHealが優先され、成功すればReborn判定はスキップ。

- 適合条件なし。DeathHealスキルを持っているだけで発動
- 蘇生後のHP: `Angel()` でε復活 → その後 `NaturalHPRecovery` で歩数ベース回復
- 戦闘後AIで既に蘇生済みなら `Death()` が false → 自動スキップ
- 戦闘後AIとの関係: 戦闘後AIは「その場で蘇生を試みて確率的に諦める」。
  本システムは「時間経過後の再会時」に改めて蘇生機会を与える **セーフティネット**。
  戦闘後AIで明示的に書けばより執拗に蘇生するキャラにもできる（性格表現）

#### b. Reborn歩数の加速

友情コンビの相方が生存している場合、死亡メンバーの `RebornSteps` による復活カウントダウンが **固定2倍速** になる。
一人で復活を待つより、仲間がそばにいる方が早く戻れる。

- 実装: `EnemyRebornManager.CanReborn` の歩数消化に2倍の倍率を適用
