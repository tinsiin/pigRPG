## 1. 全員一致（コンセンサス）

### 1.1 3層分離は正しい

```
ダメージ = Floor + SP × Scale(ATK, DEF) × Modifier
```

- Floor: 「行為の意味の保証」。SPとは独立に設計
- SP × Scale: 攻防関係 × スキル威力。成長の二軸が乗算で噛み合う
- Modifier: 精神の余裕補正。精神ダメージのみ。本体は1.0固定

SPの「二重人格」問題（旧式でSPがFloorとScaleの両方に関与）が構造的に解消される。

### 1.2 魔法ScaleはPower Ratio型で決まり

```
Scale_Magic = k × ATK^q / (ATK^q + DEF^q)
```

- 値域 (0, k) で発散しない
- ATK < DEFでもゼロにならない（A-4保証）
- if分岐不要（旧シグモイドブレンドのif矛盾を構造解消）
- q=2を全員推奨（ATK=DEFで50%、ATK/DEF≈1.3で通常とのクロスポイント）

### 1.3 余裕補正はtanh飽和

```
W(L_atk, L_def) = 1.0 + β × tanh(α × (L_atk - L_def))
```

- 値域 [1-β, 1+β] で暴走が原理的に不可能
- β=0.5前後を推奨（±50%の精神ダメージ変動）
- 劣勢でもW=0.5（β=0.5時）。精神攻撃が死に手にならない
- 旧余裕ブースト暴走（7300万ダメージ）が再発しない

### 1.4 2軸直交は離散enum

- 攻撃モデル軸（Normal / Magic）: Scale関数の形、漏れ項の有無、Floor係数を決定
- ダメージ領域軸（Body / Mental）: ATK/DEFの入力源、Modifierの有無を決定
- 連続パラメータ(μ)での補間は不採用。「÷ vs -」は質的差異であり量的グラデーションではない
- 「にじみ」はScale関数の内部（qやα）で保持する

### 1.5 統一テンプレート関数

4つの個別関数を1つのテンプレートに統合可能:

```
CalculateDamage(attacker, sp, AttackModel, DamageDomain) → StatesPowerBreakdown
```

呼び出し側:
```csharp
var model = skill.IsMagic ? AttackModel.Magic : AttackModel.Normal;
dmg       = CalculateDamage(Atker, SP,  model, DamageDomain.Body);
mentalDmg = CalculateDamage(Atker, SPm, model, DamageDomain.Mental);
```

## 2. 最大の対立点：差分型Scaleの線形発散

### 2.1 問題の概要

```
Scale_Normal = max(0, ATK - DEF)
```

ATK-DEF差が大きくなる終盤、SP×(ATK-DEF)が暴走する。

バランスブレイカーの破壊テスト結果:
- ATK=500, DEF=50, SP=200 → ダメージ = Floor + 200×450×1.79 = **160,830**
- HP1500の敵が蒸発どころの話ではない

### 2.2 「式の中で抑えろ」派（バランスブレイカー）

Scale_Normalを[0,1)にバウンドする:

```
Scale_Normal_Bounded = (ATK - DEF) / (|ATK - DEF| + C) + penetration

C = 40（飽和定数）
penetration = 0.07（ATK≤DEFでも最低保証）
```

利点:
- 通常と魔法のScaleが同じドメイン[0,1)に揃い、SPとの乗算で予測可能
- 終盤のダメージ暴走が構造的に不可能

欠点:
- A-2「ハイリターン」が弱まる。壁を越えた先の報酬が頭打ちになる
- ATK-DEF差が大きい時の「力の証明」体験が鈍る

追加の発見:
- ATK=DEFでScale=0（penetrationなしの場合）→ 同格戦でSP層が完全に死ぬ
- penetrationの導入が必須（ATK≤DEFでもScale>0にする貫通率）

### 2.3 「式の外で抑えろ」派（数理モデル + 体験設計者 + 統一アーキテクト）

Scale_Normalは線形のまま保持:

```
Scale_Normal = max(0, ATK - DEF)
```

インフレ対策は式の外で行う:
1. 十日能力の成長曲線にソフトキャップ（既存の成長距離15制限が実質的なキャップ）
2. ATK係数テーブルの正規化（Raincoat=22.0の支配問題の解消）
3. 敵のDEF設計をプレイヤーATKに追随させる
4. 補正チェーン内のソフトキャップ（最終手段）

利点:
- 差分型の「壁を越えたら報われる」線形的リターンが保存される
- FEやDQが歴史的に採用してきた手法であり実績がある

欠点:
- 式だけ見ても安全性が保証されない。数値帯の設計次第で壊れうる
- バランス調整の負荷がレベルデザイン側に移る

### 2.4 RPG歴史研究者の補足

- FE: 線形発散を意図的に許容。「この敵には通用しない」が戦術シグナルになる
- DQ: ATK/2-DEF/4の係数で数値帯を圧縮（式の中だがソフトな方法）
- 世界樹: (ATK-DEF)×スキル倍率。pigRPGと最も近い構造で、同じ発散問題を持つ
- ポケモン: A/D比で正規化（比率型に近い）。差分型の発散を回避している

### 2.5 対立の本質

この対立は「式の美しさ vs ゲーム体験」ではなく、**「安全性の責任をどこに置くか」**の設計判断。

| 選択 | 式の責任 | レベルデザインの責任 | リスク |
|------|---------|-------------------|--------|
| 式の中で抑える | 高（どんな数値でも安全） | 低 | ハイリターン体験の鈍化 |
| 式の外で抑える | 低（式は信頼する） | 高（数値帯の管理必須） | レベルデザイン不備で暴走 |

## 3. 漏れ項の設計 — 3つの案

### 3.1 案A: DEF減衰率方式（バランスブレイカー推奨）

漏れ項を独立した加算項ではなく、DEFの一部を無視する形にする:

```
有効DEF = DEF × (1 - leak_rate)
Scale_Magic = k × ATK^q / (ATK^q + 有効DEF^q)

leak_rate = 0.10（DEFの10%を無視）
```

利点:
- ScaleとLeakが統合され、「LeakがScaleを圧倒する」問題が構造的に消滅
- 概念的にクリーン（「魔法はDEFの一部を透過する」）

欠点:
- A-9「漏れ項概念」がScale関数内に吸収され、独立した概念として消える
- 「素の自分(ATK_raw)の魔力の滲み出し」という設計哲学が表現しにくい

### 3.2 案B: Power Ratio方式（数理モデル設計者推奨）

漏れ項もPower Ratio形式にする:

```
Leak = λ × SP × ATK_raw^r / (ATK_raw^r + DEF^r)

λ = 0.05〜0.15（漏れ強度）
r ≤ q（メイン項を超えない構造的制約）
```

利点:
- 漏れ項が[0, λ×SP)に有界。ATKインフレで発散しない
- r ≤ qの制約でScale_Magicとの支配関係が構造的に制御される
- ATK_raw（素の自分）を使う設計哲学が保存される

欠点:
- DEFが漏れ項にも影響する（完全なDEF無視ではなくなる）
- 計算がやや複雑

### 3.3 案C: 単純方式（現行の改良）

```
Leak = λ × SP × ATK_raw

λ = 0.02〜0.03（現行0.09から大幅に下方修正）
```

利点:
- 最もシンプル。A-9「DEFを完全に無視する浸透成分」を文字通り保存
- ATK_raw × 小さな係数で概念的に明快

欠点:
- ATKの線形成長で漏れ項も線形に伸びる。終盤でScale_Magicを圧倒するリスクが残る（現行0.09問題の根と同じ構造）
- ただしλを十分小さくすれば実用上問題ない可能性あり

### 3.4 各案の比較

| 観点 | 案A: DEF減衰率 | 案B: Power Ratio | 案C: 単純 |
|------|---------------|-----------------|-----------|
| Scale形骸化リスク | なし（統合） | なし（r≤q制約） | あり（λ次第） |
| DEF無視の忠実度 | 部分無視 | 部分無視 | 完全無視 |
| 設計哲学の保存 | 弱い | 中 | 強い |
| 実装の単純さ | 最も単純 | やや複雑 | 単純 |
| インフレ耐性 | 高 | 高 | 中 |

## 4. Floor層の設計 — 2つの案

### 4.1 案A: SP完全独立（数理モデル設計者推奨）

```
Floor = f（スキルランクごとの固定値）

基本スキル: f = 2〜4
上位スキル: f = 5〜8
TLOA:      f = 8〜12
```

利点:
- SPの二重人格問題が完全に解消。DEFの価値がSP成長で薄まることが原理的にない
- Floorの調整がScale/SP成長と独立に行える

欠点:
- ゲーム進行でFloorの相対的価値が低下（終盤HP1500に対してf=12は0.8%）
- Floor自体の成長体験がない

### 4.2 案B: SP収穫逓減型（統一アーキテクト案）

```
Floor = floor_coef × SP^p

p = 0.7〜0.8（収穫逓減）
floor_coef = スキルランクで変動
```

利点:
- SPの成長がFloorにもゆるやかに反映される。スキルレベルが上がると最低保証も微増
- 収穫逓減(p<1)によりDEFの価値が高SP帯でも保護される

欠点:
- SPとFloorが完全に独立ではない（ただしp<1で旧式のp=1.0より大幅に改善）

### 4.3 体験設計者の指針

FloorはHP比1-3%が目標。「ゼロじゃなかった」の安心感と「少なすぎて意味がない」の不満の境界。

- HP60（序盤）に対して Floor 2-3 → 3-5%。「届いた」の実感あり
- HP1200（終盤）に対して Floor 12-15 → 1%。存在は感じるが支配的ではない

## 5. テンポデザイナーの提案 — 動的要素

計算式の静的構造だけでなく、戦闘中の時間変化を設計する提案。

### 5.1 興奮の指数減衰

現行の線形補間（興奮→素へ均一に戻る）を指数減衰に変更:

```
ExcitementDecay(t) = exp(-k × t/T_max)

k = 3.0（80%の効果が最初の40%のターンに集中）
```

体験: スキル直後は興奮がほぼフル、最初の1-2ターンで急速に減衰し、残りは微かな余韻。「にじみ」の設計哲学との整合性が高い。

### 5.2 ComposureBonus — 落ち着きが精神攻撃を強化

```
精神ATK_effective = 精神ATK × (1.0 + α × (1.0 - DecayCurve(t)))

α = 0.1〜0.2（落ち着き完了時の精神ATKボーナス上限）
```

- スキル直後（興奮中）: ボーナスなし
- 完全に落ち着いた状態: 10-20%の精神ATKボーナス
- 「自分自身に帰った者だけが、相手の心の奥に手が届く」

### 5.3 PressureFactor — 戦闘中の余裕変動

Leisure能力値そのものは固定だが、「実効余裕」を戦闘状況で変動させる:

```
Leisure_eff = Leisure_base × PressureFactor

PressureFactor = HP_factor × MentalHP_factor × Momentum_factor

HP_factor       = 0.7 + 0.3 × (HP / MaxHP)        ... HP0%で30%減
MentalHP_factor = 0.6 + 0.4 × (MentalHP / MaxMHP)  ... MHP0%で40%減
Momentum_factor = 1.0 - 0.1 × min(3, RecentHitCount) ... 連続被弾で最大30%減
```

これにより余裕補正Wのtanh入力が戦闘中に動的に変化し、精神攻撃の「いつ使うか」の判断が面白くなる。

### 5.4 「体は硬くなるが心は折れる」構造

既存の引き締め値メカニクスとの組み合わせ:

```
連続被弾時:
  引き締め値 → DEF上昇（体が攻撃パターンに適応）
  RecentHitCount増加 → PressureFactor低下 → Leisure_eff減少（心が揺らぐ）
  → 本体ダメージは減るが、精神ダメージは増える
```

| 状況 | 本体攻撃の有効度 | 精神攻撃の有効度 | 最適行動 |
|------|----------------|----------------|---------|
| 初撃 | 高（DEF未適応） | 中（余裕差のみ） | 本体攻撃 |
| 2-3連続被弾後 | 低（引き締め→DEF上昇） | 高（余裕低下+被弾焦り） | 精神攻撃に切替 |
| 間が空いた後 | 中（適応リセット途中） | 低（余裕回復） | 再度本体攻撃 |

「殴り続ければ体は硬くなるが、心は削れる。だから攻め方を変えろ」という自然なフェーズ切替を促す。

### 5.5 理想的な戦闘タイムライン

```
序盤(T1-3) 「叩きつけ」
  興奮: 高 → 本体攻撃が最も強力
  余裕: 双方フル → 精神攻撃は素質差のみ
  引き締め: 未発動 → DEFは基礎値

中盤(T4-8) 「読み合い」
  興奮: ほぼ消失 → 本体攻撃力は素のATK
  余裕: 差が出始める → 精神攻撃の有効性が分岐
  引き締め: 進行中 → 同じパターンの連打は通りにくい

終盤(T9-15) 「決着」
  興奮: ゼロ → ComposureBonus最大 → 精神攻撃の最適時間帯
  余裕: 追い詰められた側が大幅低下 → 精神攻撃が決定的に有効
  引き締め: 完全適応 → 本体攻撃は通りにくい
```

## 6. RPG歴史研究者の分析結果

### 6.1 pigRPGの歴史的位置づけ

| pigRPGの特徴 | 最も近い商用RPG | 距離 |
|-------------|---------------|------|
| Power Ratio型Scale | ダークソウルのS字カーブ防御 | 近い |
| 差分型Scale + SP乗算 | 世界樹の迷宮 | 近い |
| Floor + Scaleの分離 | DQ/ポケモンの最低保証 | 中 |
| 精神HP(ATK/DEFベース) | **前例なし** | — |
| 余裕差→精神攻撃力 | **前例なし** | — |
| 「引く vs 割る」の意識的差別化 | **前例なし** | — |
| 37種十日能力の重み付き畳み込み | **前例なし** | — |
| 3層分離×2軸直交フレームワーク | **前例なし** | — |

### 6.2 歴史的に前例のない5つの設計

1. **ATK/DEFベースの精神HP計算**: 他のゲーム（Darkest Dungeon, D&D, ペルソナ）は精神軸に独自の計算体系（固定値、セーヴ判定、自発消耗）を用いる。pigRPGは本体ダメージと同じフレームワークで精神ダメージを計算し、入力パラメータだけを差し替える

2. **余裕差が精神攻撃力を決定する設計**: 「攻撃者と防御者のステータス差分が第2HP軸の攻撃力を決定する」は商用RPGに直接的前例がない

3. **「引く vs 割る」の算術レベルでの意識的差別化**: 多くのRPGが物理/魔法で別ステータスを用意する中、同じATK/DEFに対して計算式の構造だけで差別化する設計は唯一

4. **37種のサブステータスの重み付き畳み込み + 内訳追跡**: 畳み込み後にどの成分がどれだけ寄与したかを追跡するStatesPowerBreakdownは、機械学習のアテンション機構に近い発想

5. **3層分離×2軸直交フレームワーク**: 個々の要素（Floor、差分型、比率型、Modifier）は既存RPGに断片的に存在するが、2軸を直交させて4式を統一原則から導出する設計は前例なし

### 6.3 歴史的事例からの具体的提言

| 提言 | 根拠 | 理由 |
|------|------|------|
| q=1.5〜2.5で開始 | ダクソのS字カーブが実効的にq≈2 | 急すぎず緩すぎないS字遷移 |
| β=0.3〜0.5が妥当 | Darkest Dungeonのストレス抵抗が±30%程度 | 過大は暴走、過小は意味消失 |
| 漏れ項係数0.02〜0.03 | FFTの「魔法防御なし」が極端すぎた歴史 | メイン項を食わない範囲 |
| 精神用テーブルでLeisure係数を最大にしない | ポケモンの「1種族値が支配的→ビルド多様性が死ぬ」 | Raincoat=22.0の過ちを繰り返さない |

## 7. 統一アーキテクトのコード設計

### 7.1 C#疑似コード（核心部分）

```csharp
public enum AttackModel { Normal, Magic }
public enum DamageDomain { Body, Mental }

public static class DamageFormulaConfig
{
    public const float RatioQ = 2.0f;
    public const float RatioK = 10.0f;       // ScaleRatioの上限値（要調整）
    public const float FloorCoefNormal = 1.0f;
    public const float FloorCoefMagic  = 1.0f;
    public const float LeakCoef = 0.03f;
    public const float LeisureAlpha = 0.1f;
    public const float LeisureBeta  = 0.5f;
}

// 統一テンプレート
public StatesPowerBreakdown CalculateDamage(
    BaseStates attacker, float sp, AttackModel model, DamageDomain domain)
{
    var (atk, def, atkRaw) = ResolveInputs(attacker, domain);

    StatesPowerBreakdown scale = model switch
    {
        AttackModel.Normal => ScaleDiff(atk, def),
        AttackModel.Magic  => ScaleRatio(atk, def),
    };

    float floor = GetFloorCoef(model) * sp;

    StatesPowerBreakdown leak = model == AttackModel.Magic
        ? atkRaw * (DamageFormulaConfig.LeakCoef * sp)
        : StatesPowerBreakdown.Zero;

    float modifier = domain == DamageDomain.Mental
        ? LeisureModifier(attacker)
        : 1.0f;

    return (scale * sp * modifier) + leak + floor;
}
```

### 7.2 StatesPowerBreakdownとの整合性

| 項 | 追跡精度 | 手法 |
|---|---|---|
| Floor | N/A（十日能力非依存） | NonTenDayPartに格納 |
| ScaleDiff × SP | 完全 | operator-で差分→operator×でスカラー乗算 |
| ScaleRatio × SP | 近似（ATK比率保存） | Total計算→ATK比率でスケーリング |
| Leak | 完全 | operator×でスカラー乗算 |
| LeisureModifier | 完全（スカラー乗算） | float値をoperator×で全要素に適用 |

比率型ScaleRatioでは `ATK^q/(ATK^q+DEF^q)` が非線形のため、厳密な十日能力成分ごとの寄与追跡は不可能。ただしATKの内訳比率を保存してスケーリングする近似（現行のStatesPowerBreakdown除算演算子と同じ手法）で実用上十分。

### 7.3 拡張性

| 拡張 | 必要な変更 | 既存コードへの影響 |
|------|----------|-----------------|

### 9.1 差分型の発散対策を式の中でやるか外でやるか

| 選択肢 | 支持者 | トレードオフ |

## 10. 提案される4式の具体的形（数理モデル設計者案）

### 式I: 通常攻撃・本体ダメージ

```
Damage_NB = f_n + SP × max(0, A - D)
```

### 式II: 魔法攻撃・本体ダメージ

```
Damage_MB = f_m + SP × k × A^q / (A^q + D^q) + λ × SP × A_raw^r / (A_raw^r + D^r)
```

### 式III: 通常攻撃・精神ダメージ

```
Damage_NM = f_n + SPm × max(0, Am - Dm) × W(L_atk, L_def)
```

### 式IV: 魔法攻撃・精神ダメージ

```
Damage_MM = f_m + SPm × k × Am^q / (Am^q + Dm^q) × W(L_atk, L_def) + λ_m × SPm × Am_raw^r / (Am_raw^r + Dm^r)
```

### 記号一覧

```
A     = 本体ATK(SkillAttackModifier) — 興奮込み
A_raw = 本体ATK() — 素の自分
D     = 本体DEF
Am    = 精神ATK — 十日能力の精神用重みテーブルから算出
Am_raw= 精神ATK（素の自分）
Dm    = 精神DEF — 十日能力の精神用重みテーブルから算出
SP    = SkillPower（本体向け）
SPm   = SkillPowerForMental（精神向け）
f_n   = 通常Floor（スキルランク固定値）
f_m   = 魔法Floor（f_m > f_n）
k     = ScaleRatioの上限値
q     = ScaleRatioの曲線パラメータ（推奨: 2）
λ, λ_m = 漏れ項係数
r     = 漏れ項の曲線パラメータ（r ≤ q）
W     = 1.0 + β × tanh(α × (L_atk - L_def))
α     = 余裕感度
β     = 余裕最大補正幅（推奨: 0.5）
```
