## 1. 計算式の概要

### 1.1 通常ダメージ式

```
ダメージ = (ATK × SkillAttackModifier - DEF) × SP + SP
```

展開: `SP × (ATK × mod - DEF + 1)`

実装: `BaseStates.Damage.cs:917` `NonMagicDamageCalculation()`

### 1.2 魔法ダメージ式

```
ダメージ = MagicBlendVSCalc(ATK, DEF) × (SP × 0.5) + SP × ATK_raw × 0.09
```

MagicBlendVSCalcの内部:
```
weight = 1 / (1 + exp(-k × (ATK - DEF - t)))    // k=0.02, t=140

差分計算 = ATK - DEF
除算計算 = ATK / DEF + 8 + (ATK - DEF) / 1.6

ATK - DEF ≤ 0: 除算計算のみ
ATK - DEF > 0: weight × 差分 + (1 - weight) × 除算
```

実装: `BaseStates.Damage.cs:922-989` `MagicDamageCalculation()`, `MagicBlendVSCalc()`

### 1.3 精神ダメージ式

| | 通常 | 魔法 |
|---|---|---|
| 本体 | `(ATK×余裕 - MentalDEF) × SPm + SPm` | `ATK×余裕 / MentalDEF × (SPm×0.6) + SPm×0.7` |

実装: `BaseStates.Damage.cs:934-945`

## 2. RPGダメージ計算の3大モデルとの対応

RPGのダメージ計算は大きく3つの型に分類される。

| モデル | 式の形 | 代表作 | 特徴 |
|--------|--------|--------|------|
| 差分型 | `ATK - DEF` | DQ, FE, エトリアン | 直感的。ATK成長の手応えが明快。DEF > ATKでパワークリフ発生 |
| 比率型 | `ATK / DEF` | ポケモン, VP | ダメージがゼロにならない。爆発力は控えめ |
| 乗算型 | `ATK × mod / DEF` | ディスガイア, ダクソ | 柔軟だが数値インフレしやすい |

**pigRPGの位置づけ:**
- 通常式 = **差分型**。`SP × (ATK - DEF + 1)` はDQ型の拡張
- 魔法式 = **差分型と比率型のシグモイドブレンド**。商用RPGにほぼ前例がない

## 3. 通常ダメージ式の評価

### 3.1 良い点

- **最低保証が自然**: ATK = DEFでもSP分のダメージが出る。「ダメージ1」のような不自然な最低値ではなく、スキル威力に依存した最低保証
- **ATKとSPの乗算関係**: ステータス成長とスキルレベルの両方が意味を持つ健全な構造
- **直感性**: ATKを1上げればSP倍のダメージ増。プレイヤーが成長を体感しやすい

### 3.2 懸念

- **線形スケーリングの発散リスク**: ATK - DEF差に減衰（sqrt、log等）やキャップがないため、ATK >> DEFで天井知らずにダメージが伸びる。ゲーム終盤のインフレに注意が必要
- **DEFのパワークリフ**: DEF > ATK × modの場合、ダメージが急速にSPまで落ちる。DEFに1投資するとATKの1投資をSP倍打ち消す対称性があり、防御特化ビルドが強すぎる/弱すぎる両極端になりやすい
- **ただし他システムで緩和済み**: 山型分布(±22%)、物理耐性、パッシブ減衰、命中段階(Critical 1.5倍/Graze 2~35%)、TLOA減衰、慣れ補正など13段の補正チェーンが発散を部分的に制御している

## 4. 魔法ダメージ式の評価

### 4.1 設計上の魅力

#### 4.1.1 2レジーム自動遷移

1つの計算式が、ATK-DEF差に応じて自動的に最適な振る舞いを選ぶ:

- **ATK >> DEF**: 差分型に近づき、高い火力を出す
- **ATK ≈ DEF**: 比率型が支配的になり、安定したダメージを保証する
- **ATK < DEF**: 完全に比率型となり、最低保証ダメージを出す

if分岐ではなくシグモイドで滑らかに遷移するため、境界付近でのダメージ急変（不連続性）がない。

#### 4.1.2 ATK/DEF共有による差別化

多くのRPG（FF, FE, ポケモン等）は物理と魔法で別の攻撃ステ/防御ステを用意する（ATK/DEF vs MAG/RES）。
pigRPGはATK/DEFを共有したまま**計算式の構造だけで差別化**している。

これにより:
- キャラのビルドが「物理型/魔法型」のステ振りではなく、十日能力の構成で決まる
- 同じキャラが状況に応じて物理と魔法を使い分ける戦術が自然に成立する

#### 4.1.3 DEF貫通の「漏れ項」

```
+ SP × ATK_raw × 0.09
```

DEFに一切依存しない永続的なダメージ成分。どれだけ硬い相手でも`SP × ATK × 0.09`は必ず通る。
「魔法障壁を完全には防げない」というファンタジー的直感と合致する。

注目: この項は`ATK()`（SkillAttackModifier未適用のraw ATK）を使用。落ち着きカウントの平準化はメイン項にだけ効き、漏れ項は常にフルATKで計算される。「基礎的な魔力の漏れは精神状態に左右されない」という解釈で筋が通る。

#### 4.1.4 精神ダメージの除算一本化

魔法の精神ダメージ式は `ATK / MentalDEF` の除算型のみでブレンドなし。「心を直接撃つ」精神攻撃は防御を引き算する通常式より除算で「常にある程度通る」方が味がある。

#### 4.1.5 RPG史的な位置づけ

差分型と比率型をシグモイド関数でブレンドする設計は、商用RPGにおいてほぼ前例がない。
理論上は「差分型のパワークリフ」と「比率型のスケーリング鈍化」の両欠点を消せるため、RPGダメージ計算の理想解に近い形。

類似の設計思想を持つゲーム:
- **SaGaシリーズ**: 不透明で複数レジームを持つ複雑な裏計算。武器/技タイプで式が分岐
- **ポケモン**: 比率型（A/D）ベース。最低保証`+2`の設計
- **ダークソウル**: 防御の減衰率にS字カーブを使用（用途は異なるが、シグモイド的思想は共通）
- **FFタクティクス**: 魔法がDEF非依存（漏れ項の極端版）

### 4.2 懸念・リスク

#### 4.2.1 SP × 0.5のスケーリング問題

メイン項が`SP × 0.5`のため、スキルレベル上昇時の恩恵が通常式の半分。
魔法スキルのレベルアップ体感が鈍い可能性がある。漏れ項（`SP × ATK × 0.09`）が補填するが、ATKが低い序盤ではほぼ機能しない。

#### 4.2.2 シグモイド定数 t=140 のゲームバランス依存性（重要）

t=140は「ATK - DEF差がこの値のとき差分型と比率型が半々になる」閾値。

| ATK - DEF | weight | 式の性格 |
|-----------|--------|----------|
| -50 | 0.02 | 98%比率型 |
| 0 | 0.06 | 94%比率型 |
| 50 | 0.14 | 86%比率型 |
| 100 | 0.31 | 69%比率型 |
| **140** | **0.50** | **半々** |
| 180 | 0.69 | 69%差分型 |
| 250 | 0.90 | 90%差分型 |

**リスク**: ゲーム中のATK - DEF差が0~80程度に収まる場合、魔法は常に比率型で動作し、差分型には遷移しない。シグモイドブレンドの仕組みが活きず、魔法ダメージの成長感が平坦になる。

**推奨**: ゲーム全体のATK - DEF差の数値帯を先に確定し、その範囲の中央〜やや上にtを配置する。k=0.02（傾き）もtとセットで調整が必要。

#### 4.2.3 除算モデルの +8 定数

`ATK/DEF + 8 + (ATK-DEF)/1.6` の `+8` により、ATK ≈ DEFの場面で魔法のblend結果が最低8。
SP=10のとき: 魔法 ≈ 8×5 + 漏れ = 40+。通常式のATK=DEF時ダメージ=SP=10と比較して約4倍。

ATKとDEFが拮抗する場面で魔法が通常より圧倒的に強くなる。意図的な「魔法は等レベル帯で安定する」設計ならよいが、+8の値一つでバランスが大きく動くため慎重な調整が必要。

#### 4.2.4 精神魔法ダメージのゼロ除算リスク

```csharp
Atker.ATK() * mentalATKBoost / MentalDEF() * (SkillPowerForMental * 0.6f)
```

`MentalDEF()` が0に近い場合、ダメージが無限大に発散する。`MagicBlendVSCalc`にはepsilonガード（line 965）があるが、精神魔法ダメージ式にはガードがない。MentalDEF()が常に正であることのシステム保証を確認すべき。

## 5. かすり（Graze）の評価

### 5.1 計算式

```
基本かすり率 = 0.35
減衰 = max(0, 被害者EYE - 攻撃者EYE) × 0.007
最終かすり率 = max(0.02, 0.35 - 減衰)
```

範囲: 2%〜35%。実装: `BaseStates.Damage.cs:1154-1170`

### 5.2 良い点

- **2〜35%の範囲がRPG的にちょうどよい温度**: 「ミスしたけど完全に無駄にはならない」体感を自然に生む
- **EYEの二重役割**: 攻撃時は命中（supremacy計算）、防御時はかすりダメージ減衰。1つのステに攻防両面の意味がありステに無駄がない
- **魔法の1/3かすり格上げ**: 完全回避時に1/3でGrazeに格上げ。「魔法は当たりやすい」をシンプルに表現

### 5.3 懸念

- **EYE差47で下限2%に到達**: かすりを完全に殺すには大きなEYE投資が必要。回避型のキャラがかすりを無効化しにくい設計。意図的であれば問題ないが、回避特化ビルドの存在を想定するなら係数0.007の調整を検討

## 6. コンテキスト（十日能力 → 四大ステ変換）の評価

### 6.1 設計上の魅力

- **攻防非対称性**: ATKは自分の戦闘規格(Protocol)で決まり、DEFは相手の攻撃パターン(AimStyle)で決まる。格闘ゲームの読み合いをRPGに持ち込んだ独創的な設計
- **連続攻撃の防御適応**: 引き締め値（NightInkKnight）で防御が徐々に適応。「殴り続ければ慣れる」の数学化
- **精神属性ごとの慣れ分割**: キャラの個性がシステムレベルで表現されている

### 6.2 懸念

- **係数の極端なバラつき**: 共通ATK係数の`レインコート: 22.0` は次位`エノクナギ: 3.0`の7倍以上。`馬鹿: -11.0`も極端。十日能力の実数値レンジと合わせた検証が必須
- **バランス調整困難**: 排他ATK係数 × 排他DEF係数 × 十日能力の3次元の掛け算。想定外の組み合わせで破綻するリスク

## 7. 通常式 vs 魔法式 — ゲームデザイン上の役割分担

```
          高DEF相手      等レベル帯       低DEF相手
通常:     最低SPのみ     SP×(差+1)       差分爆発（天井なし）
魔法:     除算で安定     +8で厚い        SP×0.5で控えめ
```

**通常 = ハイリスク・ハイリターン。魔法 = ローリスク・ミドルリターン。**

これはRPGとして健全な役割分担。プレイヤーに「硬い敵には魔法、柔らかい敵には物理」という明快な判断軸を与える。

## 9. 棄却済み案: 動的t値（シグモイド閾値の自動調整）

### 9.1 案の概要

t=140を固定値ではなく、プレイヤーと敵のATK-DEF差から動的に算出する案。
例: `t = 直近の戦闘のATK-DEF平均` や `t = 敵DEFから導出` など。

目的: ゲームの数値帯を意識せずとも、シグモイドのブレンドが常に「良い位置」に来るようにする。

### 9.2 棄却理由: シグモイドの存在意義を自己否定する

シグモイドブレンドの価値は**ゲーム進行に伴って魔法の「性格」が変化する**こと:

```
固定t:
  序盤（ATK-DEF差 小）→ 比率型支配 → 魔法は安定型
  終盤（ATK-DEF差 大）→ 差分型支配 → 魔法は火力型
  → 成長によって同じ魔法スキルの体感が変わる
```

tを動的にすると、weightが常にブレンド中央付近（≈0.5）に張り付く:

```
動的t:
  序盤: ATK-DEF=30,  t=30  → weight ≈ 0.5（半々）
  中盤: ATK-DEF=80,  t=80  → weight ≈ 0.5（半々）
  終盤: ATK-DEF=200, t=200 → weight ≈ 0.5（半々）
  → どの段階でも同じ性格。成長感がない
```

エアコンの設定温度を常に室温に追従させるようなもの — 装置として機能しなくなる。

### 9.3 「敵ごとにtを変える」変形案も同様に棄却

敵のDEFからtを導出する案も、結果は同じ:

- 弱い敵（DEF低）→ t低 → weight ≈ 0.5
- 強い敵（DEF高）→ t高 → weight ≈ 0.5

どの敵に対しても魔法が同じ感触になり、「硬い敵には魔法、柔らかい敵には物理」という戦術判断の意味が消失する。

### 9.4 結論

tは「ゲーム全体の数値設計を1つの数字に要約したもの」。ゲームのステ成長曲線が固まればtも自動的に1つに定まる。

```
正しい手順:
  ①ステ成長曲線を設計 → ②ATK-DEF差の帯域が判明 → ③tを帯域の中央〜やや上に配置 → ④テストプレイで微調整
```

動的にする余地はなく、固定値の微調整が唯一の正道。

## 10. 4式批判的精査（断罪）

4つのダメージ計算式に対する、設計上の構造的欠陥の網羅的指摘。

対象コード: `BaseStates.Damage.cs:915-945, 961-989`

```
式①: NonMagicDamageCalculation      — 通常・本体（line 917）
式②: MagicDamageCalculation          — 魔法・本体（line 924）
式③: NonMagicMentalDamageCalculation — 通常・精神（line 936）
式④: MagicMentalDamageCalculation    — 魔法・精神（line 945）
```

### 10.1 式① 通常・本体

```csharp
((Atker.ATK(SkillAttackModifier) - def) * SkillPower) + SkillPower
```

#### 欠陥1: SPの二重人格 — 乗算係数と加算フロアの兼任

SPが「倍率」と「最低保証ダメージ」を同時に兼ねている。

```
SP × (ATK - DEF)   ← 倍率として機能
+ SP               ← 最低保証として機能
```

DEF 1000の鉄壁相手でも、SP 50のスキルで50ダメージが確定する。スキルレベルが育つほど、DEFが構造的に無力になる。DEFの存在意義をSPが踏み潰す設計。

#### 欠陥2: DEFの価値が攻撃者のSP依存

DEF+1の効果 = ダメージSP分の軽減。防御への投資価値が自分のステではなく**敵のスキルレベル**で決まるため、プレイヤーが防御投資の見返りを予測できない。

#### 欠陥3: ATK-DEF差の線形発散

差分に対する減衰（sqrt、log等）やキャップがなく、ATK >> DEFでダメージが天井なく伸びる。補正チェーンの山型分布(±22%)は割合ブレのため、大ダメージほどブレ幅が大きくなり、発散を増幅する。

### 10.2 式② 魔法・本体

```csharp
(MagicBlendVSCalc(Atker.ATK(SkillAttackModifier),def) * (SkillPower * 0.5f)) + SkillPower * Atker.ATK() * 0.09f
```

#### 欠陥4: シグモイドの中にif分岐がある（微分不連続）

line 980-983でATK-DEF ≤ 0のとき除算オンリーにハードカットしている。

```
ATK - DEF = +1 → シグモイドブレンド
ATK - DEF =  0 → 除算オンリー（if分岐）
ATK - DEF = -1 → 除算オンリー（if分岐）
```

シグモイドで滑らかに遷移させることが売りなのに、ATK=DEF境界で計算方法が不連続に切り替わる。シグモイドの存在意義と矛盾するif分岐。

#### 欠陥5: 漏れ項の隠れた二次スケーリング

メイン項（MagicBlend内）とフロア項（SP × ATK × 0.09）の両方にATKが含まれる。ATK成長時に両項が同時に上昇し、実質的にATKの二次スケーリングが発生する。

終盤のATKが高い環境では、DEFを経由するメイン項よりDEFを完全無視するフロア項が支配的になりうる。DEFというシステムを式自身が裏切る構造。

#### 欠陥6: マジックナンバー4連星

k=0.02, t=140, +8, /1.6 の4定数に設計根拠がない。コメントに「思った以上に効果ないから+10とかしとくわ」とある（結局+8）。4定数が相互に影響し合うのに全てフィーリングで決定されている。

#### 欠陥7: 2つのATK()呼び出しの意図不明

メイン項は`ATK(SkillAttackModifier)`（mod付き）、フロア項は`ATK()`（素）を使用。設計意図の説明がコード内にもドキュメントにもない。意図的な判断か書き間違いか判別不能。

### 10.3 式③ 通常・精神

```csharp
((Atker.ATK() * mentalATKBoost - MentalDEF()) * SkillPowerForMental) + SkillPowerForMental
```

#### 欠陥8: 余裕ブーストの暴走

余裕ブースト = `max(0, 攻撃者.Leisure - 被害者.Leisure) × 攻撃者.MentalHP × 0.2`

Leisure差とMentalHPの積がATKに乗算されるため、条件次第で実効ATKが数千〜数万に膨張する。

例: Leisure差30, MentalHP=100 → ブースト=600, ATK=50 → 実効ATK=30,000

MentalDEFが何であろうと、SPm乗算で天文学的ダメージが発生する。

#### 欠陥9: 式①の構造コピペ

```
式①: (ATK(mod) - DEF) × SP + SP
式③: (ATK() × boost - MentalDEF()) × SPm + SPm
```

入力が違うだけで骨格が完全に同一。「精神攻撃」という別コンセプトに対し、式の形を検討した形跡がない。差分型の欠点（パワークリフ、線形発散）を精神ダメージにもそのまま継承している。

### 10.4 式④ 魔法・精神

```csharp
(Atker.ATK() * mentalATKBoost / MentalDEF() * (SkillPowerForMental * 0.6f)) + SkillPowerForMental * 0.7f
```

#### 欠陥10: ゼロ除算ガードなし

式②のMagicBlendVSCalcには`epsilon = 0.0001f`のガードがある。式④にはない。MentalDEF()が0になった瞬間Infinityが発生する。同一ファイル内30行差での不整合。

#### 欠陥11: シグモイドブレンド不採用の理由不明

式②（魔法本体）はシグモイドブレンドで差分型と比率型を融合している。式④（魔法精神）は純除算のみ。設計判断の理由が未記載。

結果: ATK×boost >> MentalDEFのとき除算結果が制御不能に大きくなる。式②にはシグモイドによる差分型遷移で大値時の制御機構があるが、式④には制御機構が存在しない。

#### 欠陥12: フロア値が通常精神より低い

```
式③（通常精神）: + SPm × 1.0
式④（魔法精神）: + SPm × 0.7
```

本体ダメージでは「魔法はDEFに強い = フロアが高い」設計だったのに、精神ダメージでは魔法の方がフロアが低い。本体と精神で設計意図が逆転しており、理由の説明がない。

### 10.5 4式横断: 統一的設計思想の欠如

#### 欠陥13: 式ごとに設計判断がバラバラ

| 設計要素 | 式①通常本体 | 式②魔法本体 | 式③通常精神 | 式④魔法精神 |
|----------|------------|------------|------------|------------|
| 計算モデル | 差分型 | シグモイドブレンド | 差分型 | 純除算型 |
| フロア | +SP | +SP×ATK×0.09 | +SPm | +SPm×0.7 |
| ゼロ除算ガード | N/A | あり(epsilon) | N/A | **なし** |
| DEF無視の漏れ項 | なし | あり | なし | なし |
| ATK参照 | ATK(mod) | ATK(mod) + ATK() | ATK() | ATK() |
| 余裕ブースト | なし | なし | あり | あり |

4つの式が4つの異なる設計思想で構築されている。「なぜこの式にはこの要素があり、別の式にはないのか」を説明する統一原則が存在しない。

## 11. 専門家チーム議論結果

4つの専門的視点（数学設計・プレイヤー体験・バランス崩壊・統一設計）から計算式を議論した結果。

### 11.1 全員一致の問題点

以下は4視点全てが独立に同じ結論に至った問題:

| 問題 | 数学設計者 | 体験設計者 | バランス設計者 | 統一設計者 |
|------|-----------|-----------|--------------|-----------|
| **余裕ブースト暴走** | tanh飽和で原理封殺 | 成長感を破壊、パチスロ的体験 | 7300万ダメージ（24万倍オーバーキル） | 上限付き乗算に再設計 |
| **式④にブレンド不在** | 対数ratio型で統一 | — | ゼロ除算リスク解消 | 2軸直交原則から必須 |
| **MagicBlendのif分岐** | 微分不連続がシグモイドの意義を否定 | — | ratio負値の原因 | 全域シグモイドに統一 |
| **フロアの不統一** | Floor層を独立分離 | — | — | floor_coef共通化 |

### 11.2 数学設計者: 3層分離原則

全4式を以下の3層に分離することを提案:

```
Damage = Floor(SP) × Scale(ATK, DEF) × Modifier
```

| 層 | 役割 | 数学的制約 |
|---|---|---|
| **Floor層** | 最低保証 | SPの単調関数。推奨: `a × SP^p` (p=0.7で収穫逓減) |
| **Scale層** | 攻防優劣 → 無次元量[0,∞) | ATK=DEFで基準値。連続・微分可能 |
| **Modifier層** | 通常/魔法、本体/精神の差異 | 定数パラメータ。Scale層を汚染しない |

**利点**: Floor層とScale層が独立のため「SPがDEFの価値を変える」交絡が原理的に発生しない。各層が独立にバランス調整可能。

#### Scale関数の提案

| 用途 | 推奨Scale | 式 | 特性 |
|------|----------|---|------|
| 通常（物理/精神） | べき乗比率型 | `k × ATK^q / (ATK^q + DEF^q)` | 上限k。ATK=DEFで k/2。線形発散が原理的に不可能 |
| 魔法（物理/精神） | 対数比率型 | `k_m × ln(1 + (ATK/DEF)^q)` | 貫通性（DEF>ATKでも正）。飽和性（対数増加）。if分岐不要。シグモイドブレンド自体を廃止可能 |

#### 余裕ブーストのtanh型飽和

```
mentalBoost = 1 + B × tanh(c × (Leisure_atk - Leisure_def) / Leisure_ref)
```

- tanhの値域[-1,1]により、mentalBoostの値域は `[1-B, 1+B]` に自動的に収束
- clampと違い全域で無限回微分可能（角がない）
- `Leisure_ref`（基準余裕差）が唯一の調整パラメータ
- 暴走が**原理的に不可能**

### 11.3 プレイヤー体験設計者: 体感品質評価

#### 成長手応えランキング

| 順位 | 式 | 評価 |
|------|------|------|
| 1 | 式① 通常・本体 | 因果が明快。ATK+1 → ダメージSP増。ただしDEF壁の停滞期がストレス |
| 2 | 式② 魔法・本体 | 安定した成長保証。ただし「なぜこのダメージか」の直感的理解が困難 |
| 3 | 式④ 魔法・精神 | SPm項の安定感はあるが、mentalBoostの不安定さが支配的 |
| 4 | 式③ 通常・精神 | **最悪**。余裕ブーストが状況依存すぎてプレイヤーの投資が報われない |

#### 精神ダメージの体感問題

- 余裕ブースト暴走時: 「よくわからないけど凄い数字が出た」→ 再現性がなく「自分の実力」感が薄い
- 劣勢時（余裕差マイナス）: boost=0で精神攻撃が**完全な死に手**。プレイヤーの選択肢が消える
- 「勝ってる側がさらに一方的に勝つ」雪崩現象

#### 式を変えない改善案（UI/エフェクト）

1. **命中種別演出**: かすり時は数値を小さく薄く、クリティカルは大きく揺れる
2. **有効度ヒント**: 攻撃後に「効果的！」「いまひとつ」テキスト表示
3. **ダメージ予測**: スキル選択時に「予想ダメージ: 120~180」レンジ表示
4. **精神攻撃の余裕差可視化**: 余裕差が大きい時の特別演出で「なぜ効いたか」を伝達

### 11.4 バランス設計者: 壊れるケースと数値証拠

#### P0（緊急）: 余裕ブースト暴走の具体数値

攻撃者: Raincoat=20(ATK≈444), Leisure=60, MentalHP=300
防御者: VeryLow, Leisure=5, MentalDEF≈0.28

```
mentalBoost = (60-5) × 300 × 0.2 = 3300
ATK × boost = 444 × 3300 = 1,465,200

式③: (1,465,200 - 0.28) × 50 + 50 ≈ 73,260,000（精神HP 300に対し24万倍）
式④: 1,465,200 / 0.28 × 30 + 35 ≈ 156,985,745（精神HP 300に対し52万倍）
```

#### P0（緊急）: MagicBlendの除算モデルが負値を返す

ATK=50, DEF=100 のとき:
```
damage_ratio = 50/100 + 8 + (-50)/1.6 = 0.5 + 8 - 31.25 = -22.75
```

「最低保証」のための除算モデルが**-22.75**を返す。原因は `diff/1.6` 項。ATK < DEFのとき diff は負なので、diff/1.6 が +8 の保証を完全に食い潰す。**ratio計算の設計意図と論理矛盾**。

対策: ratioパスから `diff/1.6` を除去するか、`max(diff/1.6, 0)` でクランプ。

#### P1（高）: 漏れ項が平時からメイン項を超えている

ATK=100, DEF=100（互角）のとき:
```
MagicBlend = 100/100 + 8 + 0/1.6 = 9.0
メイン項 = 9.0 × SP × 0.5 = 4.5 × SP
漏れ項 = SP × 100 × 0.09 = 9.0 × SP   ← メイン項の2倍
```

ATK≈DEFの通常戦闘で漏れ項がメイン項を常に超えている。MagicBlendの存在意義が平時から形骸化。

#### P2（中）: レインコート22.0の支配力

| Raincoat値 | ATK寄与 | 全ATKに占める割合（b_b_atk=4基準） |
|------------|---------|----------------------------------|
| 5 | 110 | 96% |
| 10 | 220 | 98% |
| 20 | 440 | 99% |

Raincoat単体でATKのほぼ全てを決定する。他の十日能力の寄与が誤差レベル。

#### 優先対策リスト

| 優先度 | 問題 | 対策案 |
|--------|------|--------|
| **P0** | 余裕ブースト暴走 | キャップ導入 or tanh飽和 or 乗算→加算に変更 |
| **P0** | MagicBlend ratioの負値 | `diff/1.6`を`max(diff/1.6, 0)`にクランプ or 項を除去 |
| **P0** | 式④ゼロ除算の潜在リスク | MentalDEF()に最低保証フロア追加 |
| **P1** | 漏れ項がメイン項を常に超過 | 漏れ項係数0.09→0.02~0.03に下方修正 |
| **P2** | SPの無限線形成長 | ソフトキャップ or べき乗減衰 |
| **P2** | レインコート22.0の支配力 | 係数引き下げ(8~12) or ATK寄与上限 |

### 11.5 統一設計者: 2軸直交フレームワーク

#### 統一骨格

```
ダメージ = CoreCalc(ATK系, DEF系) × SP系 × main_coef
         + SP系 × floor_coef
         + [魔法のみ] SP系 × ATK系_raw × leak_coef
```

#### 2軸の定義

| 軸 | 切り替え対象 | 選択肢 |
|---|---|---|
| **攻撃モデル軸**（通常/魔法） | CoreCalcの構造 | 差分型 `ATK - DEF` / ブレンド型 `Blend(ATK, DEF)` |
| **ダメージ領域軸**（本体/精神） | 入力パラメータ + 余裕ブーストの有無 | 本体パラメータ / 精神パラメータ |

2軸は**直交**する（互いに干渉しない）。

#### 統一後4式

| 式 | CoreCalc | ATK系 | DEF系 | 漏れ項 |
|---|---|---|---|---|
| 通常本体 | `ATK系 - DEF系` | ATK(mod) | DEF | なし |
| 魔法本体 | `Blend(ATK系, DEF系)` | ATK(mod) | DEF | あり(leak_coef) |
| 通常精神 | `ATK系 - DEF系` | ATK()×(1+LB) | MentalDEF | なし |
| 魔法精神 | `Blend(ATK系, DEF系)` | ATK()×(1+LB) | MentalDEF | あり(leak_coef_mental) |

#### 各式の一文説明

| 式 | なぜこの形か |
|---|---|
| **通常本体** | ATK-DEF差で勝敗が決まるハイリスク構造が物理攻撃の手応えを生む |
| **魔法本体** | ブレンドでDEFに負けにくい安定型 + 漏れ項で魔法の浸透力を表現 |
| **通常精神** | 同じ差分型で精神パラメータに差替 + 余裕が精神攻撃の刃になる |
| **魔法精神** | 同じブレンド型で精神パラメータに差替 + 魔法の安定性と余裕の攻撃性を両立 |

#### 現行コードからの必要変更

| 変更 | 対象 | 内容 |
|---|---|---|
| 式④ブレンド化 | `MagicMentalDamageCalculation` | 純除算 → MagicBlend呼び出し。ゼロ除算同時解消 |
| 式④フロア統一 | 同上 | `SPm×0.7` → `SPm×floor_coef`（全式共通定数） |
| 式④漏れ項追加 | 同上 | `+ SPm × ATK() × leak_coef_mental` を追加 |
| MagicBlend if撤廃 | `MagicBlendVSCalc` | `if(damage_diff.Total<=0)`を削除、全域シグモイド |
| 余裕ブースト再設計 | `DamageOnBattle` | `ATK×boost` → `ATK×(1+clamp(boost,0,上限))` |
| MagicBlend精神版 | 新オーバーロード | `k_m, t_m`を精神用に別途定義 |
