# 効果解決方法（EffectResolutionMode）仕様書

スキル効果の解決方法を定義する仕様書。
SkillType（何をするか）とは直交する軸として、効果の処理パイプラインを制御する。

関連仕様書:
- `doc/命中回避とisBad仕様書.md` — 命中計算と良い/悪い判定の全体像
- `doc/スキル/スキル計算仕様書.md` — スキル命中率・威力計算
- `doc/スキル/スキル総合仕様書.md` — スキルシステム概要

---

## §1 概要

### §1.1 EffectResolutionModeとは

スキルの効果をどう処理するかを決める軸。BaseSkill（スキルクラス単位）に設定する。

| モード | 意味 | 命中判定 | 効果処理 |
|--------|------|---------|---------|
| **Standard** | 標準パイプライン | 各SubEffectが個別に命中判定 | SkillTypeフラグごとに標準処理 |
| **Packaged** | パッケージ化 | 単一の命中判定 | `ManualSkillEffect()` override |

```csharp
// BaseSkill.Core.cs
public enum EffectResolutionMode
{
    Standard = 0,  // 標準パイプライン
    Packaged = 1,  // パッケージ化（旧Manual1）
}
```

### §1.2 SkillTypeとの関係

SkillTypeは**What（何をするか）** — Attack, Heal, addPassive 等。
EffectResolutionModeは**How（どう処理するか）** — Standard, Packaged。

この2つは直交する軸であり、独立して機能する。

- **Packaged + SkillType=0**: ManualSkillEffectだけで完結する。SkillTypeフラグは不要。
- **Packaged + SkillType併用**: ManualSkillEffectに加え、標準パイプラインの処理も走る。
  （例: Packaged + Heal なら、ManualSkillEffectの後に標準のHeal処理も実行される）

---

## §2 Standard モード

大多数のスキルが使う標準パイプライン。各SkillTypeフラグごとに独立して命中判定・効果処理を行う。

```
スキル実行 (ReactionSkillOnBattle)
  → Attack:        ATKTypeSkillReactHitCalc → ダメージ
  → addPassive:    IsBad判定 → 敵対的/友好的経路 → 個別付与
  → Heal:          SkillHitCalc → 回復
  → ...（各SkillTypeフラグごとに独立処理）
```

詳細は `doc/命中回避とisBad仕様書.md` を参照。

---

## §3 Packaged モード

### §3.1 目的

標準パイプラインでは実現できない、以下のようなスキルのための仕組み:

1. **原子的な複合効果** — 自分にパッシブA、相手にパッシブBを同時付与（all-or-nothing）
2. **効果間の結線** — 2つのパッシブ間のパッシブリンク
3. **独自発動計算** — SkillHitCalcをoverrideしてATK/DEF比較等の独自計算

標準パイプラインでは各SubEffectが個別に命中判定されるため、
「A当たったけどB外れた」が起きうる。Packagedモードはこれを防ぐ。

### §3.2 フィールド

BaseSkill（スキルクラス単位）に定義。レベルごとではない。

```csharp
[SerializeField] EffectResolutionMode _resolutionMode;  // Standard or Packaged
[SerializeField] bool _isPackagedHostile;                // Packaged時の敵対判定
```

| フィールド | 型 | 意味 |
|-----------|-----|------|
| `_resolutionMode` | EffectResolutionMode | Standard / Packaged |
| `_isPackagedHostile` | bool | `true`: 悪い攻撃（IsReactHIT）、`false`: 良いアクション（SkillHitCalcのみ） |

### §3.3 命中判定フロー

```
Packaged + IsPackagedHostile = true（悪い攻撃）:
  IsReactHIT(attacker)  → 内部でSkillHitCalc+命中凌駕も通る（§4参照）
  → MixAllyEvade
  → ManualSkillEffect()

Packaged + IsPackagedHostile = false（良いアクション）:
  skill.SkillHitCalc(target, actor: attacker)  → スキル命中のみ
  → MixAllyEvade
  → ManualSkillEffect()
```

これは `doc/命中回避とisBad仕様書.md` §1 の原則（悪い攻撃=IsReactHIT+SkillHitCalc、
良いアクション=SkillHitCalcのみ）をそのまま踏襲している。

### §3.4 ManualSkillEffect

各スキルクラスがoverrideして独自効果を定義する仮想メソッド。

```csharp
// BaseSkill.SkillHit.cs
public virtual void ManualSkillEffect(BaseStates actor, BaseStates target, HitResult hitResult)
{
    // 空実装。各スキルがoverrideして独自効果を定義
}
```

引数の `hitResult` には命中判定の結果（Critical / Hit / Graze / CompleteEvade）が渡される。
かすり・完全回避の扱いはスキル側の責任。

### §3.5 SkillHitCalcのoverride

Packagedスキルでは `SkillHitCalc` をoverrideして独自の発動計算を入れることも可能。
敵対的（IsPackagedHostile=true）の場合、IsReactHIT内部から `skill.SkillHitCalc()` が
仮想呼び出しされるため、overrideが正しく呼ばれる。

例: RabitCatはSkillHitCalcをoverrideしてATK vs DEFの独自判定を行う。

## §4 IsReactHITとSkillHitCalcのネスト構造

命中回避とisBad仕様書では「2段階判定」として記述されているが、
実装上は**IsReactHITがSkillHitCalcをネスト呼び出し**する構造。

```
IsReactHIT() {
  EYE/AGI比較
    → 命中時:
        AccuracySupremacy() で凌駕ボーナス算出
        return skill.SkillHitCalc(凌駕ボーナス, hitResult)
    → 回避時:
        魔法なら1/3でかすり → skill.SkillHitCalc(凌駕, Graze, true)
    → 爆破型+前のめり時:
        AGI比で回避チャンス → skill.SkillHitCalc(凌駕, hitResult)
    → 完全回避:
        return CompleteEvade
}
```

パイプライン側からは **IsReactHITを1回呼ぶだけで、EYE/AGI判定 → 命中凌駕 → SkillHitCalc
の全段階が完了する**。Packagedモードの敵対的パスもこの構造をそのまま使用。

## 付録A: 設計経緯

### A.1 旧実装（Manual1）

元々はSkillType enumに `Manual1_GoodHitCalc` (1<<8) / `Manual1_BadHitCalc` (1<<9) として
実装されていた。SkillTypeは「何をするか（What）」と「どう処理するか（How）」の
2つの関心事を1つのenumに混在させていた。

### A.2 リファクタリング

EffectResolutionModeとして別軸に分離し、SkillTypeを「何をするか」だけに純化した。
旧フラグはenumから完全に削除し、ビット位置（1<<8, 1<<9）はコメントで欠番として記録している
（将来のビット衝突を防止）。

### A.3 継承の修正

RabitCat/AchillesAndTurtleは当初 `BaseSkill` を直接継承していたが、
AllySkillのID・熟練度機能を使うため `AllySkill` 継承に変更した。
