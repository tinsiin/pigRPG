# データライフサイクル仕様書

本プロジェクトでは、キャラクター・スキル・パッシブ・ステージ等のゲームデータは**テンプレート（設計時データ）とランタイムインスタンス（実行時データ）の二層構造**で管理される。この仕様書はその分離の設計思想と規約をまとめたもの。

---

## 1. 基本概念：テンプレートとランタイムの分離

### なぜ分離するか

Unity の ScriptableObject や MonoBehaviour のシリアライズフィールドは、ランタイムで書き換えるとエディタ上の元データまで汚染される。これを防ぐため、ゲーム開始時（またはエンカウント時）に `DeepCopy()` でランタイム用の独立インスタンスを生成し、以降の戦闘・歩行での状態変化はランタイムインスタンスにのみ適用する。

```
[テンプレート]                    [ランタイムインスタンス]
SO / MonoBehaviour上の          DeepCopy()で生成された
シリアライズ値                   独立インスタンス

Inspector で設定     ─────→     戦闘・歩行で変化する
変更されない                    セーブ/ロード対象（将来）
```

### テンプレート値とランタイム状態の区別基準

| 区分 | 特徴 | 例 |
|------|------|-----|
| **テンプレート値** | Inspector/SOで設定。DeepCopyでコピーされる。ゲーム中に変わらない初期値 | 基礎能力値、スキルレベルデータ、パッシブ定義、復活歩数 |
| **ランタイム状態** | DeepCopyでコピーされない。戦闘・歩行中に動的に変化する | 現在HP、パワーレベル、精神属性、戦闘記憶、個体GUID |
| **テンプレート値だがランタイムで上書き可** | DeepCopyでコピーされるが、AIやゲームロジックが動的に変更する | `_interruptCounterActive`（Inspector初期値→AI制御で変更可） |

## 2. エンティティ別のDeepCopyパターン

### 2.1 NormalEnemy（敵キャラクター）

**呼び出し元:** `GameContext.GetRuntimeEnemies()` — エンカウント時にステージの敵リストから個体を生成

**流れ:**
```
StageCut._enemyList[i]  →  enemy.DeepCopy()  →  ランタイム個体
（テンプレート）              （分離）              （戦闘で使用）
```

**コピーされるもの（テンプレート値）:**

| フィールド | 説明 |
|-----------|------|
| 基礎能力値（ATK/DEF/EYE/AGI） | `InitBaseStatesDeepCopy()` 経由 |
| 十日能力（`_baseTenDayValues`） | 辞書ごとディープコピー |
| HP / MaxHP / 精神HP | 初期値として |
| スキルリスト（`EnemySkillList`） | 各スキルを `InitEnemyDeepCopy()` |
| `_interruptCounterActive` | Inspector初期値 |
| `RebornSteps` | 復活歩数設定 |
| `_brain` | AI（SO参照コピー、共有） |
| UI設定・グラフィック | 参照コピー |

**コピーされないもの（ランタイム状態）:**

| フィールド | 理由 |
|-----------|------|
| `_aiMemory`（BattleMemory） | 個体ごとの戦闘記憶。新規生成 |
| `_bondDeltas` | 相性値変動。新個体は空の関係性から始まる |
| `_enemyGuid` | 個体識別。新規GUID発行 |
| `NowPower` | 戦闘開始時・歩行で決定 |
| `MyImpression`（現在精神属性） | エンカウント時にスキル属性で決定 |

### 2.2 AllyClass（味方キャラクター）

**呼び出し元:** `CharacterDataSO.CreateInstance()` — ゲーム開始時にテンプレートSOから個体生成

**流れ:**
```
CharacterDataSO._template  →  _template.DeepCopy()  →  SetCharacterId()  →  ランタイム個体
```

**コピー内容:** NormalEnemyとほぼ同構造（`InitBaseStatesDeepCopy()` 共通）。味方固有として `_skillALLList`（全スキル）、`ValidSkillIDList`（有効スキルID）、`EmotionalAttachmentSkillID`（思い入れスキル）等。

**補足:** UIバインディング（BattleIconUI等）はDeepCopyでは行わず、`CharacterUIRegistry` 経由で別途バインドされる。

### 2.3 BaseSkill（スキル）

**呼び出し元:** 各キャラクターのDeepCopy内から `InitDeepCopy()` / `InitEnemyDeepCopy()` / `InitAllyDeepCopy()`

**コピーされるもの:** スキルレベルデータ（`FixedSkillLevelData`）、無限スキル単位値、成長ポイント、初期レベル

**コピーされないもの:**
- `bufferSkillType`, `bufferSubEffects` — 非シリアライズのランタイム一時値
- `A/B_MoveSet_Cash` — `CashMoveSet()` で戦闘開始時にレベルデータから構築
- `ReactiveSkillPassiveList`, `AggressiveSkillPassiveList` — 戦闘中に動的追加

### 2.4 BasePassive（パッシブ）

**呼び出し元:** `PassiveManager` からIDで取得 → `DeepCopy()` でインスタンス化

**分離が必須な理由:**
> インスタンス共有するとターン処理が共有されてヤバい
> （コード内コメントより）

パッシブのターンカウンター（`DurationTurnCounter`）や歩数カウンター（`DurationWalkCounter`）は個体ごとに独立して進行する必要がある。テンプレート共有だと全キャラクターで同じカウンターが動いてしまう。

**コピーされるもの:** 名前、ID、精神属性条件、種別条件、最大重ね掛け数、持続設定値、追加HP層
**コピーされないもの:** `DurationTurnCounter`（0開始）、`_owner`/`_grantor`（付与時にバインド）、`_passiveLinkList`（動的生成）

### 2.5 Stages（ステージデータ）

**呼び出し元:** `Stages.Awake()` → `DeepCopyToRunTime()`

**流れ:**
```
StageDates（シリアライズ済み）  →  DeepCopyToRunTime()  →  RunTimeStageDates
                                                          ├ StageData.DeepCopy()
                                                          │  ├ StageCut.DeepCopy()
                                                          │  │  ├ AreaDate.DeepCopy()
                                                          │  │  └ NormalEnemy.DeepCopy()  ← 敵テンプレートもここで複製
                                                          │  └ StatesBonus.DeepCopy()
                                                          └ StageThemeColorUI.DeepCopy()
```

ステージ全体をランタイム用に複製し、歩行中の敵撃破・復活等による `_enemyList` の変化がテンプレートに影響しないようにする。

## 3. DeepCopy実装時の規約

### 新しいフィールドを追加する時

1. **そのフィールドはテンプレート値か、ランタイム状態か**を判断する
2. テンプレート値 → `DeepCopy()` にコピー処理を追加
3. ランタイム状態 → コピーしない（デフォルト値 or 別のタイミングで初期化）
4. **参照型フィールド**は、共有してよいもの（SO参照、スプライト等）以外はディープコピーする

### AIから動的変更可能なフィールドの扱い

`_interruptCounterActive` のように「Inspector初期値を持つがAIが戦闘中に変更する」パターンでは：
- DeepCopy時は**Inspector初期値をそのままコピー**する（テンプレート側は変更されないため安全）
- エンカウントごとにDeepCopyで新インスタンスが作られるため、前回戦闘でのAI変更値は自然にリセットされる

### 注意: BattleAIBrain（AI）の共有

`NormalEnemy._brain` は ScriptableObject の**参照コピー**であり、ディープコピーではない。つまり同じAI定義を持つ敵は同一SOインスタンスを共有する。このため：
- AI内にミュータブルな状態を持つ場合、`SkillActRun()` 開始時に `user` を毎回設定して個体を切り替える
- エディタではSO共有汚染検出（`_lastRunActer` チェック）で警告を出す

## 4. 現方式の限界と将来の改善候補

現在の手動DeepCopyは動作上問題ないが、フィールド追加時のコピー漏れリスクや保守コストがある。将来的に規模が大きくなり管理が困難になった場合、以下の代替手段を検討できる。

| 手段 | 概要 | 利点 | 導入コスト |
|------|------|------|-----------|
| `Instantiate(SO)` | SOに対してUnity標準のInstantiateを使う。全シリアライズフィールドが自動コピーされる | フィールド追加漏れが起きない | データクラスをSO化する設計変更が必要 |
| データ/ロジック分離 | テンプレート値を `[Serializable]` な純粋データクラスに切り出し、ランタイム状態と構造的に分離する | テンプレートとランタイムの区別が型レベルで明示される。`JsonUtility` 等で一括コピーも可 | 既存クラスの大幅な構造変更 |
| `[NonSerialized]` 徹底 | ランタイム専用フィールドに `[NonSerialized]` を厳密に付与し、`Instantiate()` でランタイム値がデフォルトに落ちるようにする | 手動コピーの大部分を排除できる | 既存フィールドの属性見直しが必要 |

**現時点では手動DeepCopyで十分機能しており、急いで移行する必要はない。** 致命的に困る場面（コピー漏れの頻発、エンティティ種別の急増等）が出てきた時に検討する。

---

## 5. 将来のセーブ/ロードとの関係（概要のみ）

現時点でセーブ/ロードは作り込まれていないが、設計上の方針として：
- **セーブ対象 = ランタイムインスタンスの状態**（テンプレートはアセットとして常に存在）
- ロード時は「テンプレートからDeepCopy → セーブされた状態を上書き復元」の流れが自然
- `_enemyGuid` 等の個体識別子は、セーブ/ロードで同一個体を紐付けるために使用（`GameContext` での `RestoreGuid` 参照）
